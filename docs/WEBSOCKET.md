# Skycom WebSocket System

> **Status**: Active — Centrifugo v5, not ActionCable. UUID-based channels with event-driven pub/sub.

---

## 1. Overview

Skycom uses **Centrifugo v5** as its WebSocket server instead of Rails ActionCable. A single persistent WebSocket connection is established per page session. Messages are published from the backend (controllers, jobs, services) and delivered to frontend subscribers using an event-envelope protocol over named channels.

Since all entity IDs in Skycom are **UUIDs**, channel names are simply the raw UUID string — no prefixing or transformation needed.

### Docker

Centrifugo runs as a service in `docker-compose.yml`:

```yaml
websocket:
  image: centrifugo/centrifugo:v5
  ports:
    - "8000:8000"
  command: centrifugo --client_insecure=false --api_insecure=false
  environment:
    CENTRIFUGO_API_KEY: "skycom_super_secret_api_key_2026"
    CENTRIFUGO_TOKEN_HMAC_SECRET_KEY: "skycom_jwt_hmac_secret_token_key_2026"
```

- **WebSocket endpoint**: `ws://127.0.0.1:8000/connection/websocket`
- **HTTP API endpoint**: `http://localhost:8000/api` (used by backend to publish)

## 3. Connection Lifecycle

```
HTTP Request
    │
    ▼
WebsocketConcern#set_websocket_channels  (before_action)
    │
    ├── WEBSOCKET.company_channel(current_company&.id)  → "<company_uuid>"
    └── WEBSOCKET.user_channel(current_user&.id)         → "<user_uuid>"
    │
    ▼
application.html.erb layout
    │
    ├── data-websocket-url-value="ws://127.0.0.1:8000/connection/websocket"
    └── data-websocket-token-value="<JWT signed by WEBSOCKET.token()>"
    │
    ▼
WebsocketController#connect()
    │
    ├── Creates one Centrifuge(url, { token }) connection
    ├── Stores as window.WEBSOCKET.cable
    ├── Exposes window.WEBSOCKET.subscribe() API
    └── Centrifugo enforces channel access via JWT claims
```

### 3.1 Backend Channel Building

`WebsocketConcern` is included in `Companies::ApplicationController` and runs `set_websocket_channels` as a `before_action`:

```ruby
# app/controllers/concerns/application_controller/websocket_concern.rb
def set_websocket_channels
  return unless is_signed_in? && @current_company
  @channels = []
  @channels << WEBSOCKET.company_channel(current_company&.id) if current_company
  @channels << WEBSOCKET.user_channel(current_user&.id)       if current_user
  @channels.compact!
end
```

### 3.2 Token Injection (Layout)

The `application.html.erb` layout injects the WebSocket URL and a signed JWT token onto the `<body>` element:

```erb
<body
  data-controller="... websocket"
  <% if is_signed_in? && @channels.present? %>
    data-websocket-url-value="ws://127.0.0.1:8000/connection/websocket"
    data-websocket-token-value="<%= WEBSOCKET.token(sub: current_user.id.to_s, channels: @channels) %>"
  <% end %>
>
```

The JWT is generated by `WEBSOCKET.token` and encodes:
- `sub` — user ID (identifies the client)
- `channels` — the list of channels this client is **authorized** to subscribe to

### 3.3 Frontend Connection

`WebsocketController` creates a **single** Centrifuge connection that lasts for the page session:

```javascript
// app/javascript/controllers/websocket_controller.js
initializeCentrifuge() {
  const centrifuge = new Centrifuge(this.urlValue, {
    token: this.tokenValue
  })
  window.WEBSOCKET.cable = centrifuge
  centrifuge.connect()
}
```

No other controller creates its own Centrifuge connection. All share the one from `window.WEBSOCKET.cable`.

## 4. Channel Naming

All channel names are **raw UUID strings** because all Skycom entity IDs are UUIDs. Channel generators enforce nil-safety and are symmetric between BE and FE.

### Backend (Ruby)

```ruby
WEBSOCKET.company_channel(company_id)  # => "<company_id>" or nil
WEBSOCKET.user_channel(user_id)        # => "<user_id>" or nil
```

Both accept **only IDs** (never objects). Callers must extract `.id` before calling:

```ruby
# ✅ Correct
WEBSOCKET.company_channel(current_company&.id)
WEBSOCKET.user_channel(current_user&.id)

# ❌ Wrong
WEBSOCKET.company_channel(current_company)
```

### Frontend (JavaScript)

```javascript
WEBSOCKET.companyChannel(companyId)  // => "<companyId>" or null
WEBSOCKET.userChannel(userId)        // => "<userId>" or null
```

### Channel Reference

| Channel | Pattern | Example | Purpose |
|---------|---------|---------|---------|
| **Company** | `"<company_id>"` | `"a1b2c3d4-..."` | Company-wide events (top-ups, invoices, alerts) |
| **User** | `"<user_id>"` | `"e5f6g7h8-..."` | User-specific events (not yet used) |

## 5. Event Protocol

All WebSocket messages use a standard envelope with a registered event name. The frontend filters incoming publications by the `event` field.

### Envelope Structure

```json
{
  "event": "top_up.completed",
  "id": "txn-uuid-here",
  "payload": {
    "amount_cents": 10000,
    "transaction_id": "txn-uuid-here"
  }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `event` | string | Dot-separated event name (must be registered in EVENTS) |
| `id` | string | UUID of the resource this event concerns |
| `payload` | object | Arbitrary data body (everything except `id` from the `data:` hash) |

### Event Registry

Both BE and FE maintain a synchronized `EVENTS` constant. **When adding a new event, you must update both.**

```ruby
# Ruby — config/initializers/websocket.rb
EVENTS = {
  test:             "test",
  top_up_completed: "top_up.completed",
  invoice_paid:     "invoice.paid",
  balance_updated:  "balance.updated",
  alert_triggered:  "alert.triggered"
}.freeze
```

```javascript
// JavaScript — app/javascript/controllers/websocket_controller.js
EVENTS: {
  test:             "test",
  top_up_completed: "top_up.completed",
  invoice_paid:     "invoice.paid",
  balance_updated:  "balance.updated",
  alert_triggered:  "alert.triggered"
}
```

| Key (BE Symbol / FE String) | Event Name | Current Subscribers | Payload Fields |
|----------------------------|------------|-------------------|----------------|
| `test` | `"test"` | None (dev only) | `{ test: true }` |
| `top_up_completed` | `"top_up.completed"` | `top_ups/new_controller.js` | `{ amount_cents, transaction_id }` |
| `invoice_paid` | `"invoice.paid"` | — | (reserved) |
| `balance_updated` | `"balance.updated"` | — | (reserved) |
| `alert_triggered` | `"alert.triggered"` | — | (reserved) |

## 6. Publishing from the Backend

Use `WEBSOCKET.publish_event` from any controller, job, or service:

```ruby
WEBSOCKET.publish_event(
  channel: WEBSOCKET.company_channel(company&.id),
  event_key: :top_up_completed,
  data: {
    amount_cents: 10_000,
    transaction_id: txn.id
  }
)
```

### Signature

```ruby
def self.publish_event(channel:, event_key:, data: {})
```

| Param | Type | Description |
|-------|------|-------------|
| `channel` | String | The target channel (use `company_channel` or `user_channel` helpers) |
| `event_key` | Symbol | Key into `EVENTS` hash — raises if unregistered |
| `data` | Hash | Payload. If `data[:id]` is present, it becomes the top-level `id` field; the rest goes into `payload` |

### Who Can Publish

| Source | File | What It Publishes |
|--------|------|-------------------|
| **Webhooks::BankPaymentController** | `app/controllers/webhooks/bank_payment_controller.rb` | `top_up_completed` to company channel after wallet credit |

## 7. Subscribing from the Frontend

Use `window.WEBSOCKET.subscribe()` from any Stimulus controller:

```javascript
const channel = WEBSOCKET.companyChannel(currentCompany().id)
WEBSOCKET.subscribe(channel, "top_up_completed", (resourceId, payload) => {
  toast({ type: "success", message: "Top-up successful!" })
  window.location.href = Helpers.company_billing_path(companyId)
})
```

### Signature

```javascript
subscribe(channelName, eventKey, callback)
```

| Param | Type | Description |
|-------|------|-------------|
| `channelName` | String | Target channel (use `companyChannel` or `userChannel` helpers) |
| `eventKey` | String | Key into `EVENTS` — rejected if unregistered |
| `callback` | Function | `(resourceId, payload) => {}` — called when an event matches |

### The `callback` receives two arguments:

1. **`resourceId`** — the `id` field from the envelope (the UUID of the resource)
2. **`payload`** — the `payload` object from the envelope (everything except `id`)

### Multiplexing Behavior

`subscribe()` reuses an existing Centrifuge subscription if one already exists for the same channel. It attaches an additional `publication` listener that filters by `event` field. This means:

- One WebSocket connection per page
- One Centrifuge subscription per channel (shared)
- Multiple event listeners per subscription (filtered by event name)

```javascript
let sub = this.activeSubscriptions[channelName]
if (!sub) {
  sub = this.cable.newSubscription(channelName)
  sub.subscribe()
  this.activeSubscriptions[channelName] = sub
}
sub.on('publication', (ctx) => {
  if (ctx.data?.event === expectedEvent) {
    callback(ctx.data.id, ctx.data.payload)
  }
})
```

## 8. Adding a New Event (Step by Step)

### Step 1: Register in BE EVENTS

```ruby
# config/initializers/websocket.rb
EVENTS = {
  # ... existing events ...
  stock_low: "stock.low",
}.freeze
```

### Step 2: Register in FE EVENTS

```javascript
// app/javascript/controllers/websocket_controller.js
EVENTS: {
  // ... existing events ...
  stock_low: "stock.low"
}
```

### Step 3: Publish from Backend

```ruby
# In a controller, job, or service:
WEBSOCKET.publish_event(
  channel: WEBSOCKET.company_channel(company&.id),
  event_key: :stock_low,
  data: {
    id: stock.id,
    product_name: stock.product.name,
    remaining_quantity: stock.quantity
  }
)
```

### Step 4: Subscribe from Frontend

```javascript
// In any Stimulus controller:
const channel = WEBSOCKET.companyChannel(currentCompany().id)
WEBSOCKET.subscribe(channel, "stock_low", (resourceId, payload) => {
  toast({
    type: "warning",
    message: `Low stock: ${payload.product_name} (${payload.remaining_quantity} left)`
  })
})
```

## 9. Full Example: Top-Up Flow

This is the production flow that demonstrates the complete publish-subscribe cycle.

### 9.1 Backend: Bank Webhook Publishes

```ruby
# app/controllers/webhooks/bank_payment_controller.rb
class Webhooks::BankPaymentController < ActionController::Base
  def create
    # ... validate signature, find transaction, credit wallet ...

    ActiveRecord::Base.transaction do
      wallet = company.billing_wallet.lock!
      wallet.update!(main_balance_cents: wallet.main_balance_cents + amount)
      txn.update!(status: :completed)
    end

    # ── Broadcast to all connected clients for this company ──
    WEBSOCKET.publish_event(
      channel: WEBSOCKET.company_channel(company&.id),
      event_key: :top_up_completed,
      data: {
        amount_cents: amount,
        transaction_id: txn.id
      }
    )

    render json: { status: "completed" }
  end
end
```

### 9.2 Frontend: Top-Up Page Subscribes

```javascript
// app/javascript/controllers/companies/top_ups/new_controller.js
renderQRWait(response, amountCents, companyId) {
  // ... render QR code for user to scan ...

  // ── Subscribe to the company channel ──
  const channel = WEBSOCKET.companyChannel(currentCompany().id)
  WEBSOCKET.subscribe(channel, "top_up_completed", (resourceId, payload) => {
    toast({ type: "success", message: translate("Top-up successful! Redirecting...") })
    setTimeout(() => {
      window.location.href = Helpers.company_billing_path(companyId)
    }, 1500)
  })
}
```
